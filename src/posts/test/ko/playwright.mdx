## 기본배경


playwright MCP가 나오기전에 playwright를 사용해서
포탈에서 발생한 150개정도의 화면 테스트 케이스에 대해서 관리를 하였다.

typescript로 발생할 수 있는 객체의 데이터 타입에 대해서 적용하였고,
객체가 변경될 경우 테스트 케이스도 개발자가 수동으로 변경하도록 설정했다.

근데 이러한 방식은 문제가 있었다.
개발자가 해당 기능에 대해 테스트 케이스를 다시 살펴보는 시간도 걸리고,
까먹는 경우나 데이터의 변경에 대해서도 하나하나 신경을 써야했기에 너무 번거로운 작업이 생겼다.

## AI 기반 QA 엔지니어링

개발자 -> 터미널에서 명령어를 쳐서 playwright를 기동시킨다 -> 테스트 코드를 수정

- 문제점 : 기능수정마다 테스트 케이스에 대한 컨벤션을 수시로 확인해야하고, 데이터가 변경될경우 데이터를 수동으로 설정하여 변경해줘야한다.

MCP의 등장

개발자 -> Claude Code에게 playwright를 기동시키고, skills를 참고하여 새로운 테스트 케이스를 작성해달라고한다.

- 장점: 기능수정이 발생할경우 Claude Code에게 부탁시켜서 skills를 참고한 데이터의 형식과 템플릿 파일을 보고 자동으로 수정을 부탁한다.


과거도 그렇고 현재도 그렇고 QA 엔지니어와, 소프트웨어 개발자의 영역이 분리되어 있다고 생각했는데
이러한 과정을 사용하면 개발자가 동시에 QA 엔지니어의 역할도 어느정도 수행할 수 있다고 생각이 들었다.


Claude에 Skills가 등장하게 되었으니
- Playwright 작성에 대한 convention TEST_GUIDE.md
- Faker 라이브러리를 사용해 ?  - Faker 라이브러리가 뭔지부터 확인해봐야할듯
- 선택자 우서순위 등을 정의핻두자.

그 이후로 playwright mcp를 기동시킬 방법을 찾아보려고 한다.

claude code 한테 playwright mcp 요청하는법

"LoginPage가 제대로 동작하는지 브라우저에서 직접 테스트해줘"
